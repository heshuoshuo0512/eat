# 智慧食堂项目技术规划与学习文档

## 一、项目定位

本项目面向高校学生，建设一个集食堂信息查询、菜品筛选、用户评价、排行榜、营养数据展示和 AI 个性化饮食推荐于一体的智慧食堂平台。

项目主要解决两个问题：

1. **校园餐饮信息不对称**
   - 学生不知道各个食堂、楼层、档口有哪些菜品。
   - 学生不知道菜品价格、口味、营业时间和真实评价。
   - 学生存在“找餐难、选餐难、不知道吃什么”的问题。

2. **校园健康饮食服务缺失**
   - 食堂菜品缺少热量、蛋白质、脂肪、碳水等营养信息。
   - 学生有减脂、增肌、维持体重、健康饮食等个性化需求。
   - 传统食堂无法根据学生身体情况和饮食目标进行推荐。

项目最终目标是：

> 构建一个基于本校真实食堂数据的智慧餐饮平台，让学生更快找到想吃的饭，更科学地选择适合自己的饮食。

## 二、项目核心功能

### 1. 食堂全景信息服务

该部分主要解决“找餐难”和“选餐难”。

#### 1.1 食堂分类导航

功能包括：

- 按食堂展示。
- 按楼层展示。
- 按档口展示。
- 展示食堂位置。
- 展示营业时间。
- 展示档口简介。

用户可以快速知道：

- 哪个食堂在哪里。
- 某个食堂有哪些楼层。
- 某个楼层有哪些档口。
- 某个档口卖什么菜。

#### 1.2 菜品多维度查询

支持按照以下条件查询和筛选菜品：

- 菜品名称。
- 菜系。
- 口味。
- 价格。
- 食材。
- 是否清真。
- 热量范围。
- 蛋白质含量。
- 减脂 / 增肌 / 健康标签。

示例筛选：

```text
15 元以内 + 高蛋白 + 微辣 + 午餐推荐
```

#### 1.3 实时排行榜

排行榜类型包括：

- 菜品排行榜。
- 档口排行榜。
- 食堂排行榜。

排行榜可以根据以下数据计算：

- 用户评分。
- 评论数量。
- 浏览量。
- 收藏量。
- 销量，后期接入。

MVP 阶段可以先使用：

```text
综合分 = 平均评分 * 0.7 + 评论数权重 * 0.3
```

#### 1.4 用户评价系统

用户可以对以下对象进行评价：

- 菜品。
- 档口。
- 食堂。

评价内容包括：

- 评分。
- 文字评论。
- 图片评论，后期实现。
- 评论时间。
- 用户昵称。

MVP 阶段建议先实现：

- 1 到 5 星评分。
- 文字评论。
- 评论列表展示。

### 2. AI 个性化营养服务

该部分主要解决“健康饮食”和“个性化推荐”。

#### 2.1 菜品营养数据库

为本校食堂中的菜品建立营养数据库。

每道菜建议包含：

- 菜品名称。
- 所属食堂。
- 所属档口。
- 价格。
- 图片。
- 主要食材。
- 口味标签。
- 是否清真。
- 热量。
- 蛋白质。
- 脂肪。
- 碳水。
- 膳食纤维，后期补充。
- 钠含量，后期补充。
- 维生素，后期补充。

MVP 阶段必须优先完成：

- 热量。
- 蛋白质。
- 脂肪。
- 碳水。

#### 2.2 用户健康档案

用户可以录入个人健康信息。

包括：

- 性别。
- 年龄。
- 身高。
- 体重。
- 运动习惯。
- 健康目标。
- 饮食偏好。
- 忌口食物。
- 预算范围。

健康目标包括：

- 减脂。
- 增肌。
- 维持体重。
- 健康饮食。

#### 2.3 智能餐单生成

系统根据以下信息生成推荐餐单：

- 用户健康档案。
- 用户饮食目标。
- 用户口味偏好。
- 用户预算。
- 当前可选菜品。
- 菜品营养数据。

推荐结果可以包括：

- 早餐推荐。
- 午餐推荐。
- 晚餐推荐。
- 推荐理由。
- 总热量估算。
- 蛋白质、脂肪、碳水估算。

#### 2.4 营养摄入追踪

后期可以支持：

- 记录用户每日吃了什么。
- 自动计算每日摄入热量。
- 自动生成周报。
- 自动生成月报。
- 提醒蛋白质不足、热量超标、脂肪偏高等问题。

## 三、技术栈规划

- uni-app。

- JavaScript。

- CSS。

- uView Plus 或 uni-ui 组件库。

前端分为：

3. 管理员端。

- 先做管理员端可以更快完成数据维护。

### 2. 后端技术栈

后端使用：

- Java。
- Spring Boot。
- MyBatis-Plus。
- MySQL。
- Redis。
- RabbitMQ。
- JWT。
- HTTPS。
- Docker。

MVP 阶段优先使用：

- Spring Boot。
- MyBatis-Plus。
- MySQL。
- JWT。
- Docker。

Redis、RabbitMQ、HTTPS 可以后期逐步加入。

#### 各技术作用

| 技术 | 作用 |
|---|---|
| Spring Boot | 后端接口开发 |
| MyBatis-Plus | 简化数据库增删改查 |
| MySQL | 存储用户、食堂、档口、菜品、评论等数据 |
| Redis | 缓存热门数据、排行榜、验证码等 |
| RabbitMQ | 异步任务、通知、数据统计等 |
| JWT | 用户登录认证 |
| HTTPS | 提升传输安全性 |
| Docker | 部署后端、数据库、Redis 等服务 |

---

### 3. AI / Agent 技术栈

AI 方向使用：

- Python。
- LangChain。
- 开源大模型。
- RAG 检索增强。
- 向量数据库。

可选向量数据库：

- FAISS。
- Chroma。
- Milvus。

建议分阶段实现 AI：

#### 第一阶段：规则推荐

不直接使用大模型，而是通过规则进行推荐。

例如：

- 减脂用户优先推荐低热量、高蛋白、低脂菜品。
- 增肌用户优先推荐高蛋白、适中碳水菜品。
- 清真用户过滤非清真菜品。
- 低预算用户优先推荐低价菜品。

#### 第二阶段：AI 生成推荐理由

后端先筛选出真实存在的菜品，再让 AI 生成推荐理由。

例如：

```text
根据你的目标是减脂，系统推荐鸡胸肉饭、清炒西兰花和番茄鸡蛋。
这些菜品整体热量较低，蛋白质含量较高，适合控制体脂。
```

#### 第三阶段：RAG + 个性化推荐

将菜品营养库、用户健康档案、饮食记录作为知识库，让 AI 结合检索结果生成更个性化的餐单。

重要原则：

> AI 只能基于数据库中真实存在的菜品进行推荐，不能凭空生成学校食堂不存在的菜。

## 八、推荐系统设计

### 1. MVP 推荐方式

MVP 阶段建议使用规则推荐，而不是一开始就完全依赖 AI。

推荐流程：

```text
读取用户健康档案
    |
读取菜品营养数据
    |
根据目标筛选菜品
    |
根据预算、口味、评分排序
    |
返回推荐菜品
```

---

### 2. 减脂推荐规则

优先推荐：

- 热量较低。
- 蛋白质较高。
- 脂肪较低。
- 非油炸。
- 少糖。

示例规则：

```text
减脂推荐分 = 蛋白质分 + 低热量分 + 低脂分 + 评分分 + 口味匹配分
```

---

### 3. 增肌推荐规则

优先推荐：

- 高蛋白。
- 热量适中或偏高。
- 碳水适中。
- 搭配主食。

---

### 4. 维持体重推荐规则

优先推荐：

- 热量适中。
- 营养均衡。
- 口味匹配。
- 价格合适。

---

### 5. AI 推荐原则

AI 不应该直接决定推荐结果，而是基于后端筛选出的候选菜品生成解释。

正确流程：

```text
数据库真实菜品
    |
规则筛选
    |
候选菜品列表
    |
AI 生成推荐理由
```

错误做法：

```text
直接让 AI 凭空生成菜品
```

因为 AI 可能会推荐学校食堂不存在的菜。

## 十七、新手团队容易踩的坑

### 1. 一开始功能做得太大

不要一开始就做完整 AI Agent、RabbitMQ、复杂推荐系统。

正确路线：

```text
先完成信息查询平台
再完成评价和排行榜
再完成营养数据库
再完成规则推荐
最后接入 AI
```

---

### 2. 前后端接口不统一

建议尽早使用：

- Apifox。
- Postman。
- Swagger。
- Knife4j。

接口先定好，前后端才能并行开发。

---

### 3. 忽视数据采集

这个项目的核心不是单纯写代码，而是建立本校真实菜品数据。

需要提前安排：

- 谁去拍照。
- 谁录入价格。
- 谁整理营养信息。
- 谁审核数据。
- 菜品多久更新一次。

---

### 4. AI 过早复杂化

AI 是亮点，但不是基础。

项目基础是：

```text
真实菜品数据 + 营养数据 + 用户健康档案
```

没有这些数据，AI 推荐没有实际价值。

---

### 5. 没有阶段性可演示版本

每个月都应该有一个可以运行的版本。

即使功能很少，也要能展示：

```text
登录 → 查看食堂 → 查看菜品 → 查看详情 → 获取推荐
```

## 二十、总结

智慧食堂项目的核心不是一开始就做复杂 AI，而是先建立一个真实、可用、可维护的校园食堂数据平台。

推荐的建设顺序是：

```text
信息查询 → 评价排行 → 营养数据库 → 规则推荐 → AI 推荐 → 数据分析与规模化
```

只要按照阶段推进，每个月都有可演示成果，这个项目就能稳步落地。
