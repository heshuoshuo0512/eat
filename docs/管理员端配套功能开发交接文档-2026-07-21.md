# 管理员端配套功能开发交接文档

更新时间：2026-07-21

## 一、范围与原则

本次工作只改造 Web 管理员端及其配套后端、数据库能力，不修改微信小程序，也不重复建设学生端推荐或 Agent 服务。管理员端复用现有评价、帖子、上传、排行、记忆、引用和可信度接口。

学生端正在由另一窗口维护，因此本工作区存在混合未提交改动。后续合并必须按文件和功能边界处理，不能整文件覆盖共享文件。

## 二、已完成内容

### 1. 导航与权限

- `/admin` 无面板参数时自动进入内容审核：`/admin?panel=reviews&tab=reviews`。
- 评价管理更名为内容审核，提供评价审核和帖子审核两个页签。
- 保留 `/admin?panel=data`、`/admin/input`、`/agent` 和 `/admin/ai` 地址。
- 内容审核使用 `review:moderate`、`post:moderate`；数据录入使用 `canteen:write`、`stall:write`、`dish:write`；运营智能体使用 `agent:use`；AI 配置使用 `ai:configure`；租户管理仅使用 `tenant:manage`。
- 未授权角色仍可进入页面，但只显示权限提示，不渲染可操作内容。

### 2. 内容审核

- 评价支持状态、评价类型、食堂、档口、菜品筛选，默认按最新待审核内容加载。
- 正确区分菜品评价和食堂评价，并展示完整的菜品、档口、食堂上下文。
- 帖子审核展示作者、正文、单图、关联对象、可选评分、审核状态和关联评价状态。
- 支持通过、驳回、重新审核，操作期间禁用按钮并显示结果反馈。
- 带评分的菜品帖子由后端事务同步正式评价，前端只展示同步结果，不重复创建评价。
- 帖子没有删除入口，帖子审核继续沿用现有评价审核体系和审计记录。

### 3. 四区数据管理

固定四个一级餐饮区及桌面位置：

| 位置 | 区域 ID | 区域 |
| --- | --- | --- |
| 左上 | `campus-main` | 中央餐饮区 |
| 右上 | `north-zone` | 北苑餐饮区 |
| 左下 | `south-zone` | 南湖餐饮区 |
| 右下 | `east-zone` | 东苑餐饮区 |

- 桌面端固定为 2×2 等宽布局，移动端改为单列。
- 缺少固定区域时展示空状态，不使用其他食堂补位。
- 区域内部层级为：食堂 → 一级档口 → 子档口 → 菜品。
- 旧档口保持一级档口身份；旧菜品可以直属一级档口，并标识为“直属菜品”。
- 每层支持展开、收起、搜索、数量统计，以及按权限显示新增和编辑入口。
- 从节点新增数据时，会携带食堂、一级档口或子档口上下文跳转到 `/admin/input`。

### 4. 数据录入工作区

- 长页面改为食堂、一级档口、子档口、菜品、批量导入五个分段任务，一次只显示一个任务。
- 顶部使用餐饮区 → 食堂 → 一级档口 → 子档口级联选择，后续实体自动继承关系。
- 菜品表单分为基础信息、营养与安全、图片与展示，高级字段折叠但不降低校验要求。
- 图片上传和 AI 图片识别整合到菜品表单；识别结果只预填，必须由管理员确认后保存。
- 提供“保存”“保存并继续新增”“保存后返回数据管理”三种明确操作。
- 连续录入时保留当前食堂和档口上下文。
- CSV 为主要批量导入方式；JSON 放在高级导入折叠区，仍采用预览后确认。

### 5. 运营智能体

`/agent` 定位为运营智能体调试页，保留任务输入、快捷任务、会话 ID、多轮追问、回答、意图、角色、执行计划和风险等级。

页面同时展示可信度、工具成功率、安全性、总耗时、真实引用和待确认操作。执行轨迹默认只展示工具、状态、风险和耗时等关键字段；工具结果只展示脱敏摘要，不显示原始 Schema、载荷、完整响应、凭证或原始 SSE 日志。本次不开放评测用例 CRUD、记忆编辑和原始事件流界面。

### 6. AI 配置与运行监控

- AI 页面拆分为提供商配置、运行监控、租户与额度三个页签。
- 连接测试只显示模型、状态和耗时，不显示长回复样例。
- API Key 留空保存时保留旧密钥；只有确认“清空配置”才删除。
- 运行监控保留月调用量、成功率、Token/图片数、平均耗时和剩余额度。
- 日志默认每页 20 条，只展示时间、功能、模型、状态、Token/图片数、耗时和不超过 80 字的失败摘要。
- 日志 ID、租户 ID、用户 ID、Provider 重复信息、原始载荷、长错误堆栈和恒为零的成本字段不展示。
- 新增“检索索引”监控区，展示文档数、失败数和索引状态，并支持管理员使用 `prune: true` 触发重建；重建后自动刷新状态。
- 部署就绪检查包含 `ai:configure` 鉴权，租户页签只对 `tenant:manage` 可见。

## 三、后端与数据库配套

- `stalls` 增加可空自关联字段 `parent_id`，前端和接口统一使用 `parentId`。
- 子档口必须与父档口属于同一租户、同一食堂；父档口必须是一级档口，禁止自关联、循环和三级嵌套。
- 删除有子档口的一级档口返回冲突；叶子档口继续沿用原删除行为，并由前端明确提示影响范围。
- 管理员档口层级迁移已同步 SQLite 初始化和 PostgreSQL 基线。
- 评价审核路由统一使用 `review:moderate`，返回完整目标上下文。
- 帖子状态、关联评价同步、排行失效和审核审计在同一事务中完成；PostgreSQL 使用同一连接绑定事务。
- 审计记录增加结构化元数据，包含审核前后状态、关联评价 ID、目标类型和目标 ID。
- AI 配置使用请求级租户隔离；租户 0 额度可正确保留，未知状态会直接拒绝。
- 数据库工作台不能绕过档口父子约束。
