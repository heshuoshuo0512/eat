# 学生端小程序接口与数据库联调总结

更新时间：2026-07-23  
适用范围：微信小程序学生端 `miniapp/src`  

## 1. 本轮交付范围

本轮把微信小程序调整为与 Web 学生端相同的业务能力，但保留微信触控交互和页面结构。小程序只通过 Node API 访问数据，不直接连接 SQLite 或 PostgreSQL。

### 前端完成内容

- 统一使用 `miniapp/src` 作为源码目录，移除根目录旧页面副本，更新 CLI 构建和微信开发者工具导入说明。
- 底部导航固定为“首页 / 找菜 / 社区 / 我的”，补齐智能推荐、拍照识餐、食堂导航、排行榜、区域推荐、菜品详情、健康档案、收藏与吃过、今日点餐预览和帖子发布等页面。
- 首页保留两步逐张揭晓、菜品检索、智能推荐、食堂导航、排行榜和区域推荐；揭晓记录只在首次揭晓时写入偏好接口。
- 找菜和智能推荐共用菜品卡、动态快捷问题、评分排序、引用摘要、可信度和记忆相关交互。
- 社区支持帖子/评价切换、食堂或菜品关联、单图发布和本人审核状态展示；评价支持菜品与食堂总览。
- 我的页面统一承载收藏与吃过、历史订单、健康档案、隐私协议和减少动画设置。
- 今日点餐只保留菜单、购物车本地预览、历史订单和取餐码查看；小程序源码没有订单创建调用。
- 增加统一的灰白表面、食堂绿强调色、固定图片比例、紧凑筛选控件、加载失败占位和减少动画规则。
- 增加缓存时间记录 `lastLoadedAt` 和 `refreshIfStale(maxAgeMs = 15000)`：页面进入时优先展示已有数据，超过 15 秒后台刷新，下拉刷新强制请求，弱网时保留缓存。

### 本轮没有做的事情

- 小程序视觉和页面重构自身没有新增后端路由、数据库表、字段或迁移文件。
- 没有修改管理端 Web 页面和小程序以外的业务逻辑。
- 没有把演示菜品混入接口结果，也没有开放 `POST /api/orders`。

## 2. 小程序使用的接口

下面是 `miniapp/src/services/apiClient.js` 当前封装的接口。接口合同以 `openapi/smart-canteen.yaml` 和 `server/app.js` 为准。

| 方法 | 路径 | 用途 | 主要页面/状态 |
| --- | --- | --- | --- |
| GET | `/api/health` | 服务健康检查 | 启动检查、部署验收 |
| POST | `/api/auth/login` | 账号密码登录 | 登录页备用登录方式 |
| POST | `/api/auth/wechat-login` | 微信 code 登录 | 登录页主登录方式 |
| GET | `/api/bootstrap` | 一次加载当前租户的用户、食堂、档口、菜品、评价、偏好和健康档案快照 | Store 初始化和刷新 |
| GET | `/api/dishes/:id` | 获取菜品详情、位置、供应和评价关联信息 | 菜品详情页 |
| POST | `/api/dishes/search` | 按关键词和上下文检索菜品 | 找菜页 |
| GET | `/api/menus/today` | 获取指定餐次的今日菜单、供应和售罄状态 | 首页、找菜、今日点餐预览 |
| GET | `/api/rankings` | 获取菜品、档口和食堂排行数据 | 首页、排行榜页 |
| GET / POST | `/api/recommend` | 自动加载或按健康档案请求推荐 | 智能推荐页 |
| POST | `/api/agent/assistant` | 执行智能问答、检索、推荐和工具编排 | 智能推荐、Agent 页面 |
| GET / PUT / DELETE | `/api/agent/memory` | 读取、保存和清理用户 Agent 记忆 | 智能推荐记忆区 |
| POST | `/api/agent/actions/:id/confirm` | 确认需要用户确认的 Agent 动作 | Agent 操作区 |
| POST | `/api/agent/actions/:id/reject` | 拒绝待确认 Agent 动作 | Agent 操作区 |
| GET / POST | `/api/reviews` | 查询公开评价、提交学生评价 | 社区、菜品详情 |
| GET / POST | `/api/posts` | 查询帖子、提交帖子 | 社区、发布帖子页 |
| POST | `/api/uploads` | 上传帖子或评价使用的单张图片 | 发布帖子页 |
| POST / PUT | `/api/health/profile` | 保存健康目标、预算、餐次、口味、营养和忌口 | 健康档案页 |
| GET / PUT | `/api/preferences/dishes` | 查询和更新收藏状态 | 收藏与吃过、菜品详情 |
| POST | `/api/preferences/dishes/:id/drawn` | 记录逐张揭晓 | 首页揭晓卡 |
| POST | `/api/preferences/dishes/:id/eaten` | 记录吃过 | 菜品卡、收藏与吃过 |
| GET | `/api/orders` | 查询本人历史订单和取餐码 | 我的、今日点餐预览 |
| POST | `/api/vision/meal-analyze` | 上传图片后进行拍照识餐 | 拍照识餐页 |

### 401 和刷新约定

API 客户端统一携带 Bearer token。收到 401 时清理本地 token，并通过 `reLaunch` 只触发一次登录跳转；普通网络失败不清空已有 Store 数据。四个 Tab 页面显示时调用缓存刷新策略，社区页签切换时重新读取帖子或评价。

### 同批合并的 feature/sql 数据契约

本次准备部署的集成分支同时合入了 `feature/sql` 已审核的数据契约，和小程序客户端改造是两个独立层次：

- `dishes.regional_taste`：保存区域推荐使用的稳定风味分类，接口字段为 `regionalTaste`；为空时前端仍可按菜系、口味、标签和食材兜底。
- `dishes.allergens_json`：保存菜品过敏原列表，推荐和检索的硬约束可以与食材字段一起判断。
- `health_profiles.allergies_json`：保存用户过敏食材，接口字段为 `allergies`，与忌口和清真偏好一起参与推荐。
- SQLite 初始化和 PostgreSQL 基线均包含这些列；已有 PostgreSQL 环境通过 `server/migrations/010_region_allergen_contract.sql` 幂等补列、回填空数组并补齐非空约束。
- 菜品管理写入和推荐/检索接口已经同步读写这些字段；本次冲突合并保留了管理员目录的租户防覆盖和层级校验。

本次实际合入的是上述三列和对应迁移，不把 `feature/sql` 分支中尚未进入集成分支的试验性汇总表或论坛表当作生产依赖。

### 现有表与页面映射

- `tenants`、`users`：租户和学生身份；登录响应带回当前用户快照。
- `canteens`、`stalls`、`dishes`：食堂、档口和菜品目录；图片、位置、价格、口味、菜系、标签、营养、过敏原和状态由这些表提供。
- `menus`、`menu_items`：日期、餐次、发布状态、供应量、售罄状态和供应时间；今日菜单优先使用已发布数据。
- `health_profiles`：饮食目标、预算、餐次、口味、饮食模式、辣度、营养关注、忌口、清真和低人流偏好。
- `user_dish_preferences`：收藏、吃过次数、揭晓次数及最近操作时间。
- `reviews`：菜品或食堂评价、评分、正文、审核状态和作者关联；公开列表只返回审核通过内容。
- `campus_posts`：帖子关联类型和目标、正文、单图、可选评分、审核状态及关联评价 ID。
- `uploads`：租户范围内的图片元数据和存储 URL；图片上传仍复用现有鉴权和 MIME 校验。
- `orders`、`order_items`、`payments`：历史订单和取餐码查询；本轮小程序只读，不创建订单。
- `agent_sessions`、`agent_messages`、`agent_memories`、`agent_actions`、`agent_eval_runs`：Agent 会话、记忆、待确认动作和运行指标。
- `rag_documents`、`ai_usage_logs`、`audit_logs`、`app_settings`：引用来源、AI 用量、审计和运行配置。

### 需要准备的正式数据集

上线前至少需要保证当前租户拥有以下数据：

1. 四个主要餐饮场所及其餐饮分区、档口层级和图片。
2. 每个有效档口的菜品：名称、价格、菜系、口味、标签、食材、营养、过敏原、图片和 `active/hidden` 状态。
3. 当日已发布菜单，以及每个菜单项的供应量、售罄状态和供应时段。
4. 学生账号、健康档案和可选的收藏/吃过记录。
5. 已审核评价和帖子；待审核内容只能在作者本人和管理端审核页可见。
6. Agent 使用的知识库文档和可追溯引用来源；没有外部模型密钥时允许使用服务端确定性 fallback。

本地联调快照曾验证：13 个食堂、12 个档口、17 道菜品、14 个今日菜单项、17 条排行记录；这些是当前开发库数量，不是生产数据要求，也不应在生产环境硬编码。

## 4. Web 管理端与小程序联调链路

两端共用同一个 Node API 和同一租户数据库，数据流如下：

```text
Web 管理端录入/审核
        |
        v
Node API + tenant_id 校验 + SQLite/PostgreSQL
        |
        +--> /api/bootstrap、/api/menus/today、/api/rankings --> 小程序首页/找菜
        +--> /api/reviews、/api/posts -----------------------> 小程序社区
        +--> /api/health/profile、/api/preferences/dishes ---> 小程序个人记录/推荐
        +--> /api/agent/*、/api/rag/search -------------------> 小程序智能推荐
```

管理端更新菜品、菜单、审核评价或帖子后，小程序在下次进入页面、超过 15 秒缓存有效期或用户下拉刷新时重新获取数据。小程序不直接读写数据库文件，也不依赖管理端前端状态。

## 6. 部署注意事项

- 生产服务器：`ubuntu@101.34.216.33`，项目目录 `/opt/smart-canteen`，PM2 进程 `smart-canteen`。
- 部署前必须保留服务器现有未提交改动和 stash，并先备份 SQLite，再执行 `PRAGMA quick_check`。
- 服务器更新时使用 `DB_MIGRATE=1`；本轮小程序改造本身不需要数据库迁移，但服务器已有迁移逻辑仍按部署流程执行。
- 微信开发者工具导入 `miniapp/dist/build/mp-weixin`；正式发布前仍需 HTTPS API 域名和微信合法域名。
