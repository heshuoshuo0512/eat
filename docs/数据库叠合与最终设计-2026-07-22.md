# 数据库分支叠合与最终设计

更新时间：2026-07-22
最终基线：`main`

## 分支审核结论

本次没有直接合并 `feature/sql` 的全部提交。该分支的部分结构会删除主分支已有能力：档口 `parent_id`、`campus_posts`、审计 `metadata_json`、RAG 检索字段和 PostgreSQL `vector(1536)` 设计。因此采用向后兼容的选择性叠合。

保留主分支：

- `campus_posts`，不新增重复的 `forum_posts`。
- `stalls.parent_id` 两级父子档口层级。
- `audit_logs.metadata_json` 审核状态和操作上下文。
- RAG `search_text`、`content_hash`、`embedding_model`、`vector(1536)`、HNSW 和 pg_trgm。
- 评价实时汇总，不新增重复的 `dish_review_summary` 真相表。

吸收并统一：

- `dishes.regional_taste`：区域推荐或管理员录入时的展示层分类，空字符串表示未指定。
- `dishes.allergens_json`：菜品过敏原数组，默认 `[]`。
- `health_profiles.allergies_json`：用户过敏原数组，默认 `[]`。

## 迁移位置

| 数据库 | 新库基线 | 已有库增量 |
| --- | --- | --- |
| SQLite | `server/database.js` 建表定义 | `server/database.js` 的幂等 `ALTER TABLE` |
| PostgreSQL | `migrations/postgres/001_initial_schema.sql` | `server/migrations/010_region_allergen_contract.sql` |

生产 PostgreSQL 执行 `DB_MIGRATE=1` 时由迁移执行器自动记录 `010_region_allergen_contract`。迁移只添加列、补默认值并设置非空，不删除表、不重写用户、菜品、评价或帖子数据。

## 最终关键表

### `dishes`

基础字段包括 `id`、`tenant_id`、`stall_id`、`name`、`price`、`taste`、`cuisine`、`ingredients_json`、`tags_json`、`halal`、营养字段、`rating`、`review_count`、`sales`、供应展示字段和状态。

本次确认的扩展字段：

| 字段 | 类型 | 默认值 | 用途 |
| --- | --- | --- | --- |
| `regional_taste` | `TEXT` | `''` | 粤菜清鲜、西南湘辣等展示分类，可由管理员录入 |
| `allergens_json` | `TEXT` | `[]` | 花生、牛奶、海鲜等过敏原 JSON 数组 |

### `health_profiles`

保留 `goal`、`budget_max`、`meal_type`、`taste`、`halal_only`、`avoid_json`、`dietary_pattern`、`spice_level`、`nutrition_focus_json`、`prefer_low_crowd`、`favorite_tags_json`。

新增确认字段：

| 字段 | 类型 | 默认值 | 用途 |
| --- | --- | --- | --- |
| `allergies_json` | `TEXT` | `[]` | 用户明确的过敏原，推荐硬过滤，不自动放宽 |

## 数据链路

1. `rowToDish` 输出 `regionalTaste` 和 `allergens`。
2. `rowToProfile` 输出 `allergies`。
3. 健康档案保存接口写入 `allergies_json`，保存后立即刷新推荐。
4. 普通规则推荐和 Agent 推荐都把 `avoid + allergies` 与菜品食材、`allergens_json` 比较。
5. 未设置字段的旧记录按空数组/空字符串处理，旧接口字段不失效。

## 需要准备的数据集

- 菜品基础集：每道菜的真实 `stall_id`、价格、餐次、口味、菜系、食材、标签、营养、清真、图片和供应状态。
- 过敏原集：按菜品填写标准化词，如 `花生`、`牛奶`、`虾`；没有确认来源时填 `[]`，不能猜测。
- 区域分类集：可选填 `regional_taste`；未填时前端继续依据现有菜系、口味、标签和食材分类。
- 用户档案集：`allergies_json` 与 `avoid_json` 使用 JSON 字符串数组，空值统一为 `[]`。
- 社区与评价集：帖子必须关联真实食堂或菜品，评价和审核状态使用现有 `reviews`、`campus_posts` 表。

## 不采用的设计

本次不新增 `forum_posts`、`dish_review_summary`、新的区域表或新的 Agent 推荐服务，避免出现两套帖子、评价和推荐数据真相。
