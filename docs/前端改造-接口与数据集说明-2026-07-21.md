# 前端改造、接口与数据集说明

更新时间：2026-07-21

本文总结本次管理员端配套改造以及与学生端共享的检索、评价、帖子和 Agent 能力。微信小程序不属于本次前端改造范围。

## 1. 前端新增与调整

### 管理员工作台

- `/admin` 默认进入“内容审核”，支持评价审核和帖子审核页签、状态筛选、目标对象筛选、分页、通过、驳回和重新审核。
- `/admin?panel=data` 固定展示四个餐饮区：中央餐饮区、北苑餐饮区、南湖餐饮区、东苑餐饮区。桌面端为 2×2，移动端为单列；区域缺失时展示空状态，不用其他区域补位。
- 数据层级调整为“食堂 → 一级档口 → 子档口 → 菜品”。节点支持展开、收起、搜索、数量统计和按权限新增/编辑。
- `/admin/input` 改为分段录入工作区，包含食堂、一级档口、子档口、菜品和批量导入任务；支持级联上下文、AI 图片识别预填、CSV 预览确认、JSON 高级导入以及三种保存结果。
- `/agent` 定位为“运营智能体”，显示意图、角色、计划、风险、耗时、可信度、工具成功率、引用、关键轨迹和待确认动作；工具结果只显示脱敏摘要。
- `/admin/ai` 拆分为提供商配置、运行监控、租户与额度。API Key 留空保存会保留旧值，显式清空才删除；日志分页为每页 20 条并截断失败摘要。
- AI 运行监控新增检索索引状态区，展示文档数、失败数、向量数、索引版本和状态，并支持管理员触发索引重建。

### 学生端共享前端能力

- 菜品检索和智能推荐保持为两个独立请求与状态，不再复用同一响应对象。
- 菜品检索展示可解释匹配原因、供应状态、可下单状态、引用来源和降级原因。
- 推荐页面进入时只自动加载一次结构化推荐，后续快捷提问才调用 Agent；推荐结果保留餐品引用、硬约束和放宽建议。
- 评价、帖子、上传和点餐操作继续使用现有学生页面；管理员端只接入审核和同步结果，不在前端重复创建正式评价。
- 保留减少动画模式和移动端响应式布局，不新增前端依赖。

## 2. 新增或扩展接口

以下接口均使用现有认证机制。管理员接口需要对应权限，跨租户数据按当前用户租户隔离。

### 检索与推荐

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `POST` | `/api/dishes/search` | 菜品检索；支持关键词、预算、餐次、口味、清真、忌口、食堂/档口和排序筛选，返回供应状态、匹配原因和引用。 |
| `POST` | `/api/recommend` | 根据健康档案生成结构化推荐、餐单、引用和约束放宽建议。 |
| `GET` | `/api/recommend` | 保留旧版推荐兼容响应。 |
| `GET` | `/api/rag/search` | 知识/菜品引用检索。 |
| `GET` | `/api/admin/retrieval/status` | 管理员查看索引版本、文档数、向量数、失败数、运行状态和降级信息。 |
| `POST` | `/api/admin/retrieval/reindex` | 管理员重建索引；前端默认发送 `{ "prune": true }`。 |

### 评价、帖子与上传

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/reviews` | 评价总览；支持 `targetType`、`canteenId`、`stallId`、`dishId`、`sort`、`limit`、`offset`，返回作者及食堂/档口/菜品上下文和总数。 |
| `POST` | `/api/reviews` | 学生提交评价，默认进入待审核状态。 |
| `GET` | `/api/admin/reviews` | 管理员按状态、类型和目标对象筛选审核评价。 |
| `PATCH` | `/api/admin/reviews/:id/status` | 管理员通过、驳回或重新审核评价。 |
| `GET` | `/api/posts` | 返回公开帖子，并附带当前用户自己的审核状态。 |
| `POST` | `/api/posts` | 发布文字或文字加单图帖子；必须关联食堂或菜品，菜品帖子可带 1–5 分。 |
| `GET` | `/api/admin/posts` | 管理员查看待审核和历史帖子。 |
| `PATCH` | `/api/admin/posts/:id/status` | 管理员通过、驳回或重新审核帖子；带评分帖子通过后由后端同步正式菜品评价。 |
| `POST` | `/api/uploads` | 复用现有上传接口，保存租户、所有者、文件类型、大小、存储键和公开 URL。 |

### 管理员数据与运营

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/admin/database/overview` | 四区数据管理概览，返回固定区域、食堂、档口、子档口和菜品统计。 |
| `GET` | `/api/admin/database/entities` | 数据工作台实体查询，保留层级上下文和 `parentId`。 |
| `POST` | `/api/admin/canteens` | 新增食堂。 |
| `POST` | `/api/admin/stalls` | 新增一级档口或子档口；校验同租户同食堂、禁止循环和三级嵌套。 |
| `POST` | `/api/admin/dishes` | 新增菜品，支持营养、安全、图片和菜单字段。 |
| `POST` | `/api/admin/dishes/import/preview` | CSV 导入预览。 |
| `POST` | `/api/admin/dishes/import/confirm` | CSV 导入确认。 |
| `POST` | `/api/admin/dishes/vision-import` | AI 图片识别预填菜品字段，结果必须人工确认。 |
| `GET` | `/api/admin/ai-settings` | 读取脱敏后的 AI 提供商配置。 |
| `PUT` | `/api/admin/ai-settings` | 保存 AI 配置，空 API Key 不覆盖旧密钥。 |
| `DELETE` | `/api/admin/ai-settings` | 显式清空 AI 配置。 |
| `POST` | `/api/admin/ai-settings/test` | 测试连接，仅返回模型、状态和耗时。 |
| `GET` | `/api/admin/ai-usage` | 分页返回精简 AI 使用日志和统计。 |
| `GET` | `/api/deployment/readiness` | 返回 Agent、运行时、数据库和 `ai:configure` 部署检查。 |
| `GET` | `/api/admin/tenants` | 租户和额度管理，仅 `tenant:manage` 可见。 |

### Agent 与用户数据

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `POST` | `/api/agent/assistant` | 结构化 Agent 对话，返回回答、意图、工具结果、计划、引用、风险和待确认动作。 |
| `POST` | `/api/agent/stream-run` | 运营智能体流式执行。 |
| `GET` | `/api/agent/events` | 查看指定会话关键事件快照。 |
| `GET/PUT/DELETE` | `/api/agent/memory` | 用户记忆读取、更新和清空；管理员页面不开放原始记忆编辑。 |
| `GET` | `/api/agent/actions` | 查看当前用户动作中心。 |
| `GET/POST` | `/api/agent/eval-cases` | 评测用例接口，学生端不可访问。 |
| `PUT` | `/api/health/profile` | 保存健康档案并刷新推荐条件。 |
| `GET/PUT` | `/api/preferences/dishes` | 收藏、吃过和菜品偏好。 |

## 3. 需要准备的数据集与字段

### 基础租户与权限数据

- `tenants`：`id`、`name`、`status`、`plan`、`ai_quota`、`storage_quota_mb`。
- `users`：用户、租户、角色和登录凭证；管理员角色至少配置 `review:moderate`、`post:moderate`、`canteen:write`、`stall:write`、`dish:write`、`agent:use`、`ai:configure`。
- `audit_logs`：审核前后状态、目标类型、目标 ID、关联评价 ID 等结构化元数据。

### 餐饮目录数据

- `canteens`：四个固定区域关联的食堂，包含 `id`、名称、位置、营业时间、人流等级、标签、描述和图片。
- `stalls`：`id`、`canteen_id`、`parent_id`、楼层、名称、类别、评分、均价、营业状态和描述。`parent_id` 为空表示一级档口；子档口必须和父档口属于同租户同食堂。
- `dishes`：`stall_id`、名称、价格、口味、菜系、配料、标签、餐次、清真、过敏原、热量、蛋白质、脂肪、碳水、图片、状态、销量和评分统计。旧菜品可以直属一级档口。
- `menus/menu_items`：日期、餐次、发布状态、菜品、供应量、售罄状态、供应开始和结束时间；推荐和点餐依赖这些字段判断可供应性。

### 内容审核数据

- `reviews`：用户、`target_type`（`dish`/`canteen`）、目标 ID、评分、正文、审核状态和时间。只有通过的评价进入公开列表和排行。
- `campus_posts`：租户、用户、关联类型和目标 ID、正文、单图、可选评分、审核状态、关联评价 ID、创建和更新时间。帖子通过/驳回与关联评价必须在同一事务中同步。
- `uploads`：所有者、文件名、MIME 类型、大小、租户隔离存储键和公开 URL。图片上传限制沿用现有安全校验。

### 推荐与检索数据

- `health_profiles`：饮食目标、预算、餐次、口味、清真、忌口及营养偏好，用于生成推荐约束。
- `rag_documents`：`tenant_id`、`source_type`、`source_id`、标题、正文、搜索文本、元数据、内容哈希、索引时间、Embedding 模型和 1536 维向量。
- 健康知识库：按租户提供可追溯来源、标题、正文和分块信息；每个引用必须能回溯到真实来源。
- PostgreSQL 生产环境需要 `pgvector` 和 `pg_trgm` 扩展；检索迁移会创建向量、Trigram、HNSW、租户唯一约束和索引运行记录。

### Agent 与运行监控数据

- `agent_sessions`、`agent_messages`、`agent_actions`：会话、回答、工具步骤、风险等级、确认状态和动作审计。
- `ai_usage_logs`：时间、功能、模型、状态、输入/输出 Token、图片数、耗时和精简错误摘要。
- `app_settings`：AI 提供商配置和脱敏状态；密钥不得进入前端响应、日志或引用。

## 4. 联调和上线前检查

1. 先导入四个固定区域、食堂、档口层级和至少一批带营养/过敏原字段的菜品。
2. 准备一个管理员账号及完整权限，分别验证评价审核、帖子审核、父子档口录入、AI 配置和索引重建。
3. 准备至少一条待审核评价、一条待审核帖子和一条带评分菜品帖子，验证通过、驳回、重新通过和防重复评价。
4. SQLite 初始化和 PostgreSQL 迁移都执行；生产 PostgreSQL 必须确认 `pgvector`、`pg_trgm` 和 1536 维向量列可用。
5. 完成索引重建后检查 `/api/admin/retrieval/status` 的文档数、失败数、版本和状态。
6. 执行 `npm test`、`npm run build` 和 `git diff --check`，再进行管理员登录、静态资源和移动端回归。
