# 管理员端数据库变化说明

## 结论

本次管理员目录 UI 提交没有新增一套独立目录数据库，也没有改变现有菜品、评价和订单表的主键关系。管理员端复用现有租户、餐饮场所、档口、菜品和审计模型。

## 已使用的结构变化

### `canteens`

- `parent_id`：固定餐饮场所到餐饮分区的父子关系，外键指向 `canteens(id)`。
- `canteen_type`：`primary` 表示顶层餐饮场所，`sub` 表示餐饮分区。
- 建立租户和父级索引，旧扁平食堂数据继续可读。

### `stalls`

- `parent_id`：可空自关联字段，用于兼容历史子档口。
- 建立 `(tenant_id, parent_id)` 索引。
- 新建档口不再创建新的子档口；历史子档口可迁移为餐饮分区直属档口。

### `campus_posts`

保存 `tenant_id`、`user_id`、`target_type`、`target_id`、`content`、`image_url`、`rating`、`status`、`linked_review_id`、`created_at`、`updated_at`，并建立租户状态和用户时间索引。

### `audit_logs`

- `metadata_json`：结构化记录审核前后状态、目标类型、目标 ID、关联评价 ID、排行/同步结果等。

## 迁移文件

由 `server/migrations.js` 执行并记录 `schema_migrations`，`DB_MIGRATE=1` 时启用：

- `005_campus_posts.sql`
- `006_admin_stall_hierarchy.sql`
- `007_admin_audit_metadata.sql`
- `008_retrieval_pgvector.sql` 及检索运行状态表

SQLite 在 `server/database.js` 中执行等价初始化和旧库回填；PostgreSQL 迁移使用幂等 DDL，失败回滚，重复启动不会重复建表或索引。

## 数据校验和事务

- 父档口必须与子档口同租户、同餐饮分区，禁止自关联、循环和三级嵌套。
- 有子档口的一级档口删除返回冲突；叶子档口沿用删除行为。
- CSV/JSON 批量导入使用事务，任意一行失败会回滚整批。
- 帖子审核、关联评价同步、排行失效和审核审计使用同一事务连接。
- 菜品写入后同步/失效检索文档和排行缓存。
