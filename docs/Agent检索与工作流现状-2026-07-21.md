# 智慧食堂 Agent、双检索与部署现状 - 2026-07-21

## 今天完成了什么

今天围绕“菜品查询”和“智能推荐”完成了检索底座、Agent 路由、前端接线和部署准备。两条能力保持分离：菜品查询只负责真实菜品和实时供应，智能推荐负责结合健康档案给出个性化备选。

## 已落地能力

### 双检索工作流

- `POST /api/dishes/search`：规则优先解析价格、餐次、食堂、档口、口味、食材、过敏原和营养条件。
- 菜品查询执行精确匹配、中文词法检索、pgvector 语义召回和 RRF 融合。
- 价格、库存、售罄、供应时段、菜单发布状态、菜品状态和租户隔离由数据库真值控制。
- 隐藏、停用、归档菜品不会进入查询或推荐候选。
- 只有规则、精确和词法都无法判定时，才调用可选 LLM 输出结构化过滤补充；结果经过 schema 和 Zod 校验，失败自动降级。
- `POST /api/recommend`：先加载健康档案、长期偏好、历史交互、天气、拥挤度和当前餐次，再执行过敏原、忌口、清真、预算、餐次和可售状态硬约束。
- 推荐默认返回 3 个独立备选；只有明确要求搭配或套餐时才计算总价受限组合。
- 无菜单或全部售罄时返回 `orderable:false` 的菜品库参考，不伪装成当前可下单推荐。
- 菜品证据和健康知识证据分组返回。

### Agent 工作流

- 新增意图：`dish_search`、`meal_recommendation`、`knowledge_qa`。
- 新增工具：`dish.search`、`meal.recommend`、`knowledge.search`。
- 订单查询只走 `orders.mine`，运营问题只走授权分析工具，推荐问题才加载档案、菜单和推荐服务。
- `steps` 只记录真实执行的工具，不展示未执行的模型计划。
- 下单动作必须匹配用户点名且当前可售的菜品，并继续保留高风险确认。
- 推荐和菜单来自同一套排序结果，避免页面推荐与 Agent 推荐分裂。

### PostgreSQL、pgvector 与索引

- Compose PostgreSQL 镜像切换为 `pgvector/pgvector:pg17`。
- 启动时硬校验 `vector`、`pg_trgm`、`vector(1536)`、中文 trigram GIN 和 HNSW 索引。
- 文档 ID、唯一约束和查询均包含租户；向量查询按 `embedding_model` 隔离。
- 增加 `search_text`、`content_hash`、`indexed_at`、`embedding_model` 和运行记录表。
- 菜品修改增量更新文档；菜单库存变化实时联表，不重新生成 embedding。
- 增加幂等重建脚本：

  ```powershell
  node --env-file=.env scripts/reindex-retrieval.mjs
  ```

- 管理端 AI 运行监控页现在展示索引状态、文档数、向量数、失败数、更新时间、版本，并提供手动重建。

## 技术选型结论

- 正式底座：PostgreSQL + pgvector。
- 离线开发和单测降级：SQLite + 规则/词法检索。
- 核心编排：普通 JavaScript 服务、Zod、现有 `aiProvider.js`。
- 本轮不使用 LangChain 或 LangGraph；旧的失效模块、依赖和静态可用性测试已移除。
- LangGraph 只有在未来需要暂停恢复、复杂循环、跨工具重试或人工审批编排时再评估。
