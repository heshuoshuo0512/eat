# 餐品与食堂数据录入指南

## 1. 目标

健康知识库配置完成后，下一步是录入项目自己的真实业务数据，使系统能够回答：

- 哪个食堂、哪个档口可以买到餐品；
- 菜品当前价格、餐别和营养信息是什么；
- 今天是否供应、是否售罄；
- 推荐是否基于真实菜单和用户约束；
- 健康问答是否能够引用健康知识和真实菜品事实。

健康知识库负责健康解释、安全提示和饮食原则；食堂、档口、菜品、菜单和订单数据必须由项目服务端维护。

## 2. 正确录入顺序

必须按以下顺序录入：

```text
食堂 → 档口 → 菜品 → 今日菜单 → 发布菜单 → 学生端验证
```

原因：

- 档口必须关联已存在的食堂；
- 菜品必须关联已存在的档口；
- 菜单必须关联已存在的食堂和菜品；
- 只有已发布、未售罄的菜单，才能代表“今天可以买到”。

## 3. 第一步：录入食堂

### 建议字段

- 食堂 ID
- 食堂名称
- 位置
- 营业时间
- 食堂描述
- 校区、类型和父级食堂关系（如有）
- 营业状态、拥挤度和更新时间（实时运营阶段维护）

### 接口

```http
POST /api/admin/canteens
```

所需权限：

```text
canteen:write
```

食堂必须先建立，后续档口才能使用它的 `canteenId`。

## 4. 第二步：录入档口

### 建议字段

- `canteenId`：所属食堂 ID
- 档口名称
- 楼层
- 品类
- 平均价格
- 营业状态
- 档口描述

### 接口

```http
POST /api/admin/stalls
```

所需权限：

```text
stall:write
```

示例：

```json
{
  "canteenId": "north",
  "floor": "1F",
  "name": "元气蛋白餐",
  "category": "健康轻食",
  "rating": 4.8,
  "avgPrice": 16,
  "open": true,
  "description": "鸡胸肉、鱼肉、豆制品和杂粮饭为主。"
}
```

## 5. 第三步：录入菜品

### 必填业务字段

- `stallId`
- `name`
- `price`
- `taste`
- `cuisine`
- `ingredients`
- `tags`
- `nutrition`

### 推荐完整字段

```json
{
  "id": "dish-chicken-bowl",
  "stallId": "n-protein",
  "name": "香煎鸡胸杂粮饭",
  "price": 16,
  "taste": "黑椒",
  "cuisine": "轻食",
  "ingredients": ["鸡胸肉", "糙米", "西兰花", "玉米"],
  "tags": ["高蛋白", "低脂", "减脂推荐"],
  "halal": false,
  "mealTypes": ["lunch", "dinner"],
  "nutrition": {
    "calories": 486,
    "protein": 38,
    "fat": 11,
    "carbs": 58
  },
  "fiber": 4.2,
  "sodium": 520,
  "sugar": 3,
  "calcium": 45,
  "iron": 2.8,
  "allergens": ["大豆"],
  "imageUrl": "https://example.com/dish.jpg",
  "description": "主食、蛋白和蔬菜比例均衡。"
}
```

### 接口

单条录入：

```http
POST /api/admin/dishes
```

批量 JSON 导入：

```http
POST /api/admin/dishes/import
```

CSV 预览与确认导入：

```http
POST /api/admin/dishes/import/preview
POST /api/admin/dishes/import/confirm
```

### CSV 表头建议

```text
id,stallId,name,price,taste,cuisine,ingredients,tags,halal,mealTypes,calories,protein,fat,carbs,imageUrl,description
```

校验规则包括：

- `stallId` 必须属于当前租户且已存在；
- `price` 必须大于 0；
- 菜名、口味、菜系不能为空；
- 食材和标签不能为空；
- 热量、蛋白质、脂肪、碳水必须是非负数字；
- 导入前先调用 `preview`，修正错误后再调用 `confirm`。

## 6. 第四步：录入今日菜单

菜品进入菜品库后，还需要配置实际供应菜单：

- 日期；
- 餐别：早餐、午餐、晚餐等；
- 所属食堂；
- 菜单状态：草稿、已发布、已归档；
- 菜品价格（菜单价格可能与基础菜品价格不同）；
- 供应上限；
- 已供应或已售数量；
- 售罄状态；
- 供应时间。

接口：

```http
POST /api/admin/menus
```

建议先保存草稿，核对菜品、价格和供应量后再发布。

必须区分以下三种状态：

1. 菜品库中存在；
2. 今天菜单中供应；
3. 当前仍可购买。

历史菜单、未发布菜单和已售罄菜品不能当作当前可购买事实。

## 7. 营养和过敏原数据要求

健康知识库中的 WHO、FDA、CDC 等资料不能直接替代校内菜品营养实测数据。

每道菜的营养数据建议绑定：

- 标准份量；
- 重量和单位；
- 热量、蛋白质、脂肪、碳水；
- 膳食纤维、钠、糖、钙、铁等扩展营养；
- 数据来源；
- `estimated` 或 `verified` 状态；
- 置信度；
- 生效日期；
- 版本；
- 录入人和审核人。

过敏原必须区分：

- 已确认存在；
- 已确认不存在；
- 未知；
- 可能发生交叉接触。

`unknown` 不等于“无过敏原”。未确认配方或存在交叉污染风险时，系统应提示用户向档口确认，不应输出“可以放心吃”。

## 8. 建议的首轮试运行范围

不要一开始录入全校所有数据。建议先完成一个最小真实闭环：

1. 录入 1 个真实食堂；
2. 录入 2–3 个真实档口；
3. 每个档口录入 5–10 道真实菜品；
4. 创建当天的早餐、午餐或晚餐菜单；
5. 发布菜单并设置供应量；
6. 用学生账号验证展示和推荐。

## 9. 验证清单

### 数据展示

- [ ] 食堂名称、位置和营业时间正确；
- [ ] 档口归属正确；
- [ ] 菜品名称、价格、图片和描述正确；
- [ ] 餐别和标签正确；
- [ ] 营养字段单位和份量明确。

### 推荐与问答

- [ ] 推荐只使用当前租户数据；
- [ ] 预算过滤正确；
- [ ] 清真、忌口和餐别约束生效；
- [ ] 推荐结果不会把历史或售罄菜品当作可购买菜品；
- [ ] 健康问答能够引用健康知识库来源；
- [ ] 菜品事实仍来自服务端数据库，而不是外部 RAG 文档。

### 运营状态

- [ ] 菜单日期正确；
- [ ] 菜单发布状态正确；
- [ ] 供应量和售罄状态正确；
- [ ] 价格与实际售卖价格一致；
- [ ] 档口临时关闭时不会继续给出可购买暗示。

## 10. 示例数据的处理

当前项目仍包含 `src/domain/seedData.js` 中的开发示例食堂、档口和菜品。

生产环境使用前应明确区分：

- 开发环境：可以保留种子数据用于测试；
- 生产环境：导入真实租户数据；
- 示例营养值、销量、评分和价格：不得直接当作真实运营数据。

## 11.9 资料状态说明

本次资料集使用以下状态：

- `verified_page`：本次已读取并核验页面或摘录；
- `public_entry`：发现公开入口，但未下载全文或未逐条核验；
- `restricted`：需要注册、授权或当前访问受限；
- `internal_required`：必须由项目真实数据生成。

当前仍需后续人工核验的重点包括：中国居民膳食指南正式版本、中国现行食品安全国家标准、学校具体食品安全制度、校医/急救和投诉渠道、NIH ODS 具体事实表，以及运动营养原文适用范围。

## 11.10 与 RAG 导入的关系

服务端健康知识库导入器会读取 `data/health-knowledge-bases/` 下可检索的 Markdown，并将切块写入 `rag_documents`，引用元数据包括来源文件、来源 URL、资料状态、chunk 序号和 citation。

健康资料进入 RAG 后只用于一般健康解释和安全提示；以下内容仍不得由健康 RAG 代替：当前价格、库存、菜单发布状态、售罄状态、订单、权限、用户过敏与明确忌口、学校实时制度。

## 12. 与健康知识库的边界

健康知识库负责：

- 健康饮食原则；
- 营养概念解释；
- 过敏和食品安全提示；
- 减脂、增肌和均衡饮食的一般说明；
- 医疗边界和转介提示。

业务数据库负责：

- 食堂、档口和菜品；
- 当前价格和库存；
- 今日菜单和售罄状态；
- 订单和履约；
- 用户权限、忌口和授权档案；
- 校园实际制度和联系方式。

完整链路应为：

```text
健康知识库 + 真实菜品库 + 今日菜单 + 用户档案
                    ↓
          服务端规则过滤与推荐
                    ↓
       带来源引用的健康解释和可购买建议
```
